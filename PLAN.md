# 项目计划书：CS2 操盘检测与预测

> 目标读者：项目所有者 + 未来的开发会话。现状见 HANDOFF.md，本文只谈"接下来怎么走"。
> v2：已并入项目所有者对全部开放问题的答复（2026-07-19）。
> 核心决策已定：**交易永远由用户手动操作，系统只做数据、信号与记录**；最终产品形态是 App（详见阶段 D）。

---

## 0. 终极目标与阶段拆解

**终极目标**：系统能在操盘的不同阶段给出可操作的决策——
- 拉盘**开始前/早期**提示买入（吃到主升段）；
- 拉盘**进行中**识别处于剧本哪个阶段（拿得住、不被洗下车）；
- **出货期**提示卖出（逃顶,对已持仓的人最值钱）。

| 阶段 | 名称 | 一句话目标 | 状态 |
|---|---|---|---|
| A | 数据与标注完备 | 标注可信、数据不断档 | **进行中** |
| B | 检测强化 | 规则/统计层把"正在操盘"看准 | 部分完成 |
| C | 预测模型 | 从"识别现在"到"预判将来" | 未开始 |
| D | 决策闭环与产品化 | 信号→建议→流水复盘→App | 未开始 |
| E | 多用户与商业化 | 远期,不急 | 仅记录 |

**原则（每个阶段都适用）**
1. 样本不够就不上 ML,规则+统计顶着;上 ML 也从小模型开始(逻辑回归/GBDT),不碰深度学习。
2. 每个特征/阈值都要有数据依据(重跑 `scripts/analyze-manipulation-features.mjs` 说话),不拍脑袋。
3. 外部事件(版本更新等)是最危险的混淆源,标注时必须写明事由,建模时单独作为一类对照。
4. 时间序列任务一切按时间切分训练/验证,严禁未来数据泄漏。
5. **系统不碰交易执行**,只出信号和记录,买卖永远用户手动。

## 阶段 A：数据与标注完备

**A1. 清完待审核标注**（人工,项目所有者）—— **范围已重新定义,实质完成**
- 2026-07-20 项目所有者判断:buy_price=0(开箱所得,购入渠道未知)的饰品没内幕消息没法判断是不是操盘,不用审。已把这类持仓整体移出追踪范围(`lib/tracked-items.ts`),/anomalies 默认也不展示这类饰品的待审核事件——不是清空数据库,是从"要不要审"这件事上彻底排除。
- 排除之后,可审核(buy_price>0)的待审核队列已经是 0。之后的常态就是"每小时扫描 56 个 buy_price>0 持仓 + 观察池,新增异常照旧走 /anomalies 审"。
- 完成标准:pending(可见部分)清零 ✅,之后保持"每周过一遍新增异常"的节奏。

**A2. 常驻数据采集**（开发,最高优先级）—— **机制已上线（2026-07-19），30 天无断档验证中**
- 现在价格同步挂在 dev server 里,电脑不开/服务不跑就断数据,断档直接毁时序特征。
- **实际方案:Windows 原生生产构建 + 登录自启**（`scripts/start-collector.cmd`,3210 端口,登录自启via启动文件夹)——独立于开发环境,开机即跑,数据落同一个 SQLite。**Docker 已放弃**:Windows bind mount 下 SQLite 多进程写锁不可靠,原计划的 Docker 常驻服务改成本机原生进程;Dockerfile/compose 保留但只留给将来部署 Linux 服务器用,本机不装不跑容器。
- 顺路调研更多数据渠道(成交量、挂单深度、成交笔数):SteamDT/C5 的未用接口、其他公开源;找到就加进采集器,找不到就靠现有接口高频抓在售量自己攒。
- **2026-07-20 有进展**:SteamDT 价格接口本来就有返回 `biddingPrice`/`biddingCount`(求购价/求购数),之前同步代码只存了在售侧、求购侧被丢弃——已补上(迁移 `013_add_bidding_depth.sql`),不算新数据源,是把已经在拉的字段存下来。从这天起开始积累,历史数据补不了;C5 直连价格和 K 线回填都没有这两个字段。**仍然缺的**:在售侧的挂单深度(不同价位挂了多少单,不是总在售数)、成交笔数、C5 平台自己的挂单簿数据。
- 完成标准:采集器连续 30 天无断档;"量"类数据至少有一路稳定入库(✅ 求购数据这一路已达成,继续找在售深度/成交笔数)。

**A3. 数据面扩容 = 观察池**（开发一次性改造 + 项目所有者日常使用）
- 已定:不做单独的"市场面板",**观察池就是扩容入口**——项目所有者会持续把关注的单品加进观察池。
- 开发侧配套:把 K 线回填、异常扫描、嫌疑分的覆盖范围从"仅持仓"扩到"持仓+观察池"(加入观察池时自动回填 90 天历史),让新加的品立刻有完整数据。
- 对照组问题(防"模型学成识别你会买什么"):鼓励往观察池加一部分"确定没庄的正常品"做对照。

## 阶段 B：检测强化（把"正在操盘"看准）

**B1. 嫌疑分校准闭环**
- A1 标完后重跑特征分析,更新 manipulation-score 阈值/权重;以后每次大批量标注后都重跑。
- 重新验证联动特征(coMove)——千瓦/手套标完就有验证材料。

**B2. 操盘剧本验证（本阶段核心,直通预测和逃顶）—— 第一版已完成,见 `REPORT-B2.md`**
- 项目所有者给出的做盘剧本(待数据证实/修正):
  1. **低位横盘**:单品处于价格较低阶段; ✅ 已验证,是全程占比最大的状态
  2. **吸货期**:庄家长时间吸货,价格小幅上涨; ⚠️ 价格形态上和纯横盘分不开,需要量/挂单数据才能验证
  3. **会员进场**:庄家把消息散给会员,会员拿货,价格再抬一级; ❌ 现有数据看不出这个子阶段
  4. **洗盘/砸盘**:为压住拿货成本,拉高一小段后砸盘压价,诱更多小散入场; ✅ 已验证,指纹清晰(深回撤>15%接急拉)
  5. **主拉升**:正式拉盘; ✅ 已验证,指纹清晰(急拉簇,涨速中位数12.69%/天)
  6. **出货**:抛售离场。⚠️ 价格形态和普通回调区分不开,需要量数据佐证
- 方法(`scripts/analyze-playbook-stages.mjs`):138 个窗口按同饰品间隔合并成 115 个可分析 episode,对窗口前14天~后5天做分段线性拟合+BIC 无监督变点检测,再用 k-means 对涨速/波动率/回撤聚类,全程不用剧本先验,聚类完才对号入座防循环论证。
- 结论:洗盘和主拉两个指纹已经可靠,**已实现上线**（`lib/signals/washout.ts` + `lib/anomaly-scan.ts` 的 `washout_signal`）;吸货期、会员进场、出货期三个都卡在缺量/挂单深度数据——这把 A2 遗留的"调研成交量/挂单深度数据源"提级为阻塞项。**2026-07-20 有进展**:发现 SteamDT 接口本来就有求购价/求购数（`biddingPrice`/`biddingCount`），之前被同步代码丢弃，已补上开始入库（见 A2）。这不是完整方案（只是求购侧，没有在售深度/成交笔数），但足够开始重新验证吸货/出货指纹了——只是数据要从 2026-07-20 起重新攒，等几周有积累量再重跑。
- 后续:剩余 212 条待审核标完 + 求购数据积累几周后,重跑本脚本(带上 bidding 特征)让三个已验证簇的边界更稳,同时再看能否补出吸货/出货指纹。

**B3. 联动预警上线**
- 规则:某收藏品上级确认异动/高嫌疑 → 自动对同收藏品下级(炼金料)生成"可能跟涨"提示,进异常提醒页。

**B4. 嫌疑分接入自动扫描**
- 每小时给全部跟踪品打嫌疑分,高分(≥60)自动落提醒;异常页支持按嫌疑分排序。
- 低价品(<¥0.5)分数打折,治最小报价单位造成的虚高。

## 阶段 C：预测模型

**C1. 数据集构建 + C2. 基线模型与评估 —— 第一版已完成,见 `REPORT-C1-C2.md`（结论:不满足上线门槛）**
- 方法(`scripts/build-prediction-baseline.mjs`):样本=(饰品,小时)时点,标签=未来 N 天(1/3/7)内是否进入操盘窗口;特征=B1 验证过的 8 个特征 + 洗盘信号 2 个特征(联动特征/品类静态特征受限于当前标注结构,还没能力度加进去);逻辑回归(手写梯度下降,没装新依赖);按 2026-06-29 切分训练/测试(约78/22)。
- 名义结果:6 组(3 个 N × 2 套候选池)全部"打赢"随机基准和 coMove 规则基准。**但不建议上线**,三个问题:
  1. 候选池只用 53/131 个已标注/跟踪饰品自己的历史,不是全市场随机样本,数字比真实上线场景乐观(报告里用 narrow vs full 两套池对比直接量化了这个偏差有多大,PLAN 风险4 说中了)。
  2. coMove 规则基准本身是废的(当前标注集里 AUC 只有 0.50,且真正有收藏品结构的候选——千瓦/手套——因为 A1 范围调整永久失去了标注机会),打赢它没有验证意义。
  3. **最该关注**:模型给外部事件(版本更新等)打的分几乎跟真实操盘一样高,明显分不清"操盘"和"异动"——这正是 PLAN 原则3 警惕的混淆源,当前特征集测的是"异动强度",测不出"是不是有人在操盘"。
- 下一步:等全市场候选池、求购深度数据积累、能分开操盘/外部事件的特征,再重跑;C3(回测)在这些问题解决前没有意义,暂不做。

**C3. 回测**
- 模拟策略:信号买入→持有到出货信号或固定 N 天→卖出,算**扣除 C5 和悠悠有品手续费**后的净收益,对比"什么都不做"和纯规则。
- 已定:只按 C5/悠悠有品两个平台的费率算(实际交易也在这两处手动进行);低价品还要扣流动性折价(挂单深度差,名义价卖不掉)。

**C4. 上线**
- 每小时对全部跟踪品打预测分,top 候选进"机会"页,每条附触发理由。

## 阶段 D：决策闭环与产品化

**D1. 逃顶信号**
- 直接复用 B2 的"出货期指纹"做操盘结束检测;对已持仓且处于操盘中的品出"建议了结"提示。

**D2. 交易流水与月度盈利**（已定需求,可与 B 并行开发）
- 库存同步发现商品减少(卖出)时:先尝试从 C5 卖家订单接口拉真实成交价(接口已接过,`lib/api/c5` 有卖单类型);拉不到就弹窗问用户卖价。
- 记录完整流水:每笔买入价、卖出价、时间、平台;按月汇总买入额/卖出额/已实现盈利。
- 流水同时是复盘素材:信号的实盘胜率、平均收益,定期对账反哺标注和阈值。
- 配套改造:现在"卖出=直接删行"要改成"标记已售+进流水",不然卖出历史留不下来。

**D3. 产品形态:App**
- 已定目标:**最终做成电脑端+手机端 App,最低限度是网页版**。
- 演进路径(不推倒重来,沿现有 Next.js 走):
  1. 现在:本机网页(已有);
  2. 部署成常驻服务(配合 A2 的常驻采集器,手机浏览器局域网/外网可访问;本机是 Windows 原生进程,Docker 留给将来部署 Linux 服务器);
  3. PWA 化:手机"添加到主屏幕"即近似原生 App,并解锁 **Web Push 通知**——高优先级信号(联动预警、逃顶、预测高分)推到手机;**已完成第一版(2026-07-20)**:manifest+sw.js+VAPID 推送,目前接了嫌疑分预警(≥60)和同收藏品联动预警两类;逃顶信号(D1)、预测高分(C3)还没做,做出来后要补进 `lib/anomaly-scan.ts` 的 `pushNotifications` 列表。设置页 `/settings` 订阅/取消订阅/测试推送。
  4. 需要更原生再套桌面/移动壳(Tauri/Capacitor),Web 代码复用。
- 通知渠道即 App 推送,不再单独接微信/邮箱。

## 阶段 E：多用户与商业化（远期,仅记录,不急着做）

- 若开放给更多人用:用户登录体系、每用户独立的持仓/观察池/标注、数据隔离;订阅收费。

## 主要风险

1. **样本量**:138 个窗口对分类器仍偏少,且集中在 2026 年 4-7 月单一市场环境——大更新(如 5-22)一来规律可能整体改写。缓解:持续标注 + 外部事件单列 + 定期重训。
2. **数据断档**(A2 不做,后面全部白搭)。
3. **API 依赖**:SteamDT/C5 改接口、收紧配额都会伤数据管道;kline 等实测行为与文档不符,升级要重新实测。
4. **自我实现偏差**:标注大量来自"已购=操盘"规则,模型可能学成"识别你会买什么"。缓解:观察池里放一批与买入决策无关的对照品(A3)。
5. **剧本先验偏差**:B2 验证时避免"按剧本切数据再用数据证明剧本"的循环论证——分段先用无监督变点检测切,再对照剧本命名。
